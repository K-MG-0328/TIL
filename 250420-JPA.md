## 객체 연관관계와 식별자 연관관계의 차이 및 장단점 
### 객체 연관관계  
JPA에서 엔티티 간 연관관계를 객체 지향적으로 표현하는 방식입니다. 엔티티 필드에 직접 연관된 엔티티 객체를 참조합니다.  

**발생할 수 있는 경우의 수**  
**1:N**  
Store 엔티티에서 @OneToMany로 Menu 참조하고 Menu 엔티티에서 @ManyToOne로 Store 참조 (일대다 양방향)  
Store 엔티티에서만 @OneToMany로 Menu 참조 (일대다 단방향)  
Menu 엔티티에서만 @ManyToOne로 Store 참조 (다대일 단방향)  

**1:1**
Order 엔티티에서 @OneToOne로 Delivery 참조하고 Delivery 엔티티에서 @OneToOne로 Order 참조(일대일 양방향)  
Order 엔티티에서만 @OneToOne로 Delivery 참조(일대일 단방향) 반대로 Delivery 엔티티에서만 @OneToOne로 Order 참조(일대일 단방향)  

**N:M**
User 엔티티에서 @ManyToMany로 Coupon 참조하고 Coupon 엔티티에서 @ManyToMany로 User 참조(다대다 양방향)  
User 엔티티에서만 @ManyToMany로 Coupon 참조 반대로 Coupon 엔티티에서만 @ManyToMany로 User 참조(다대다 단방향)  


### 각각의 경우에서 발생할 수 있는 문제점은? 어떻게 설계해야할까?
**1:N 단방향**은 Store에서 Menu를 참조하지만, FK는 Menu 테이블에 위치합니다. JPA는 Store 저장 시 Menu의 FK를 update하는 추가 쿼리를 발생시키므로 비효율적입니다. → **다대일(N:1) 단방향 또는 양방향**을 우선 고려합니다.

**1:1 양방향**은 어느 쪽에 FK를 둘지 결정해야 하며, 주로 더 자주 조회되는 쪽에 FK를 둡니다. 단방향이라면 FK가 있는 쪽이 무조건 주인입니다.

**N:M**은 JPA가 자동으로 조인 테이블을 만들지만, 조인 테이블에 추가 컬럼(예: 발급 일시, 사용 여부)을 둘 수 없고 쿼리 추적이 어렵습니다. → **조인 테이블을 별도 엔티티(`UserCoupon`)로 승격**시켜 1:N - N:1로 푸는 것이 정석입니다.

예시)  
```java
// Store 엔티티
@Entity
public class Store {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToMany(mappedBy = "store", fetch = FetchType.LAZY)
    private List<Menu> menus = new ArrayList<>();
}

// Menu 엔티티
@Entity
public class Menu {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "store_id") // store_id는 메뉴 테이블에 생성되는 외래키 컬럼명.
    private Store store;
}
```

### `mappedBy`는 무엇을 의미하나요?
양방향 연관관계에서 **연관관계의 주인**(외래키를 관리하는 쪽)이 누구인지 JPA에 알려주는 속성입니다. `mappedBy = "store"`는 "이 컬렉션은 Menu 엔티티의 `store` 필드에 의해 매핑된 것이므로, FK 관리는 그쪽에서 한다"는 뜻입니다.

원칙:
* FK가 있는 쪽이 **주인**(`@JoinColumn` 사용)이고, 반대편이 `mappedBy`를 가집니다.
* 주인이 아닌 쪽에서 컬렉션에 객체를 추가해도 DB에는 반영되지 않습니다.
* 양방향 매핑 시에는 양쪽 객체에 모두 값을 세팅하는 편의 메서드를 두는 것이 안전합니다.

```java
public void addMenu(Menu menu) {
    this.menus.add(menu);
    menu.setStore(this); // 양쪽 동기화
}
```

### Lazy Loading과 Eager Loading은 어떤 차이가 있나요?
| 구분 | Lazy (지연 로딩) | Eager (즉시 로딩) |
|---|---|---|
| 시점 | 연관 엔티티에 실제로 접근할 때 쿼리 실행 | 부모 엔티티 조회 시 즉시 함께 조회 |
| 기본값 | `@OneToMany`, `@ManyToMany` | `@ManyToOne`, `@OneToOne` |
| 장점 | 필요할 때만 로드 → 성능 유리 | 항상 함께 로드되어 코드가 단순 |
| 단점 | LazyInitializationException 위험 | N+1 문제·불필요한 조인 발생 |
| 실무 권장 | **모든 연관관계를 LAZY로 명시**하고 필요 시 fetch join | 가급적 사용하지 않음 |

`@ManyToOne`과 `@OneToOne`도 명시적으로 `fetch = FetchType.LAZY`로 변경해두는 것이 안전합니다.

### N+1 문제는 무엇이고 어떻게 해결하나요?
부모 엔티티 N개를 조회한 뒤, 각 부모의 연관 엔티티를 조회하기 위해 N개의 추가 쿼리가 발생하는 현상입니다.

```java
// Store 10개 조회
List<Store> stores = storeRepository.findAll();   // SELECT * FROM store (1번)
for (Store s : stores) {
    s.getMenus().size();   // 각 Store마다 SELECT * FROM menu WHERE store_id = ? (10번)
}
// 총 1 + 10 = 11번 쿼리 (N+1 문제)
```

**해결 방법:**

1. **Fetch Join (JPQL)** — 한 번의 조인 쿼리로 가져옴
```java
@Query("SELECT s FROM Store s JOIN FETCH s.menus")
List<Store> findAllWithMenus();
```

2. **`@EntityGraph`** — 메서드 단위로 어떤 연관을 함께 가져올지 선언
```java
@EntityGraph(attributePaths = {"menus"})
List<Store> findAll();
```

3. **`@BatchSize`** — IN 절로 묶어서 한 번에 조회 (1 + 1 또는 1 + ⌈N/size⌉번 쿼리)
```java
@OneToMany(mappedBy = "store", fetch = FetchType.LAZY)
@BatchSize(size = 100)
private List<Menu> menus = new ArrayList<>();
```

주의: fetch join을 컬렉션(1:N)에 사용할 때는 데이터가 곱집합으로 늘어나므로 페이징과 함께 사용할 수 없습니다. 페이징이 필요하면 `@BatchSize`나 `@EntityGraph`를 활용합니다.

### `cascade`와 `orphanRemoval`의 차이는?
* **`cascade`**: 부모에 대한 영속성 작업(persist, merge, remove 등)을 자식에게 전파합니다.
* **`orphanRemoval = true`**: 부모의 컬렉션에서 자식이 제거되면(또는 부모가 삭제되면) 그 자식 엔티티 자체가 DB에서 삭제됩니다.

```java
@OneToMany(mappedBy = "store", cascade = CascadeType.ALL, orphanRemoval = true)
private List<Menu> menus = new ArrayList<>();

// orphanRemoval = true 일 때
store.getMenus().remove(menu);  // → DELETE FROM menu WHERE id = ?
```

`orphanRemoval`은 자식 엔티티의 라이프사이클이 **완전히 부모에 종속된 경우**에만 사용해야 합니다. 다른 곳에서 같은 자식을 참조하면 의도치 않은 삭제가 일어날 수 있습니다.
