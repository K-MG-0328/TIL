### main 브랜치에서 pr 요청하는 방법

	• 메인 브랜치에서 작업 후 커밋 푸시
	• pr을 위해 temp-pr 브랜치를 생성하고 원격으로 푸시
	• 깃허브에서 GitHub에 가서 Base : temp-pr <- compare : main으로 Pull Request 생성

#### git checkout -b temp-pr
-b 옵션은 새로운 브랜치(temp-pr)를 생성(create)하면서 동시에 이동(checkout)

#### git push origin temp-pr
temp-pr 브랜치를 원격 저장소(GitHub, GitLab 등)에 업로드하는 명령어

#### git branch
로컬 브랜치 목록과 현재 나의 브랜치가 무엇인지 아는 명령어

### 커밋 이전 버전으로 되돌리는 단계별 방법

#### git log --oneline --graph --all
커밋 로그 확인하는 방법

#### git reset --hard HEAD~1
	•	HEAD~1: 한 단계 전 커밋으로 이동
	•	HEAD~2: 두 단계 전으로 이동
	•	더 이전 커밋으로 가고 싶다면 HEAD~3, HEAD~4 등 사용 가능
	
#### git reset --hard efgh456
특정 커밋으로 되돌리는 경우

#### git push origin temp-pr --force
원격 저장소에 반영 (강제 푸시)

## 깃 명령어

#### git checkout temp-pr
temp-pr 브랜치로 이동

#### git reset --hard main
temp-pr 브랜치에서 해당 명령어를 사용한다면 main 브랜치의 최신 커밋을 덮어 씌운다.

	
#### git clone
git clone을 실행하면 현재 터미널에서 위치한 디렉토리(폴더)에 저장된다.

git clone https://github.com/spring-projects/spring-boot.git ~/Projects/spring-boot

## Merge vs Rebase
| 구분 | merge | rebase |
|---|---|---|
| 결과 히스토리 | 분기와 합류가 그대로 남음 | 일직선으로 정리됨 |
| 커밋 해시 | 유지 | 재작성 (해시 바뀜) |
| 안전성 | 높음 (이미 푸시된 브랜치 OK) | 푸시된 브랜치에는 위험 |
| 사용 예 | 공유 브랜치 통합 | 로컬 작업 정리, PR 전 정돈 |

```bash
# main의 최신 변경을 feature 브랜치에 가져오기
git checkout feature
git merge main           # merge commit 생성
# 또는
git rebase main          # feature 커밋들을 main 위로 재배치
```

**원칙: 이미 원격에 푸시된 브랜치는 rebase하지 말 것.** 다른 사람이 그 브랜치를 받았다면 히스토리가 어긋납니다.

## Conflict 해결 흐름
```bash
# 1. merge나 rebase 중 충돌 발생
git status                       # 어떤 파일이 충돌했는지 확인
# 2. 파일을 열어 <<<<<<<, =======, >>>>>>> 마커를 직접 편집
# 3. 충돌 해결 후
git add <해결한 파일>

# merge의 경우
git commit                       # merge commit 완성

# rebase의 경우
git rebase --continue            # 다음 커밋 적용
git rebase --abort               # 처음부터 취소하고 싶을 때
```

## 자주 쓰는 고급 명령어
**git stash** — 작업 중인 변경사항을 임시 저장
```bash
git stash                  # 현재 변경 임시 저장
git stash list             # 스태시 목록
git stash pop              # 가장 최근 스태시 적용 + 제거
git stash apply stash@{1}  # 특정 스태시 적용 (제거 안 함)
git stash drop stash@{0}   # 특정 스태시 제거
```

**git cherry-pick** — 다른 브랜치의 특정 커밋만 가져오기
```bash
git cherry-pick <commit-hash>
git cherry-pick a1b2c3..d4e5f6   # 범위 지정
```

**git reflog** — 잃어버린 커밋 복구의 비상구
```bash
git reflog               # HEAD 이동 기록 (하드 리셋 후에도 7일~기본값까지 남음)
git reset --hard HEAD@{2}
```

## reset vs revert vs restore
| 명령어 | 동작 | 히스토리 변경 | 사용 시점 |
|---|---|---|---|
| `git reset --soft <commit>` | HEAD만 이동, 변경은 staging에 유지 | O | 커밋만 취소하고 다시 묶고 싶을 때 |
| `git reset --mixed <commit>` (기본) | HEAD 이동, staging 비우기, 작업 트리는 유지 | O | 스테이징 취소 |
| `git reset --hard <commit>` | 모두 되돌림 | O | 로컬에서만, 변경 다 버려도 OK |
| `git revert <commit>` | 해당 커밋의 반대 커밋을 새로 만듦 | X (새 커밋) | 푸시된 커밋 되돌리기 |
| `git restore <file>` | 작업 트리의 파일을 HEAD 상태로 되돌림 | X | 단일 파일 변경 취소 |

**원격에 이미 푸시된 커밋은 reset 대신 revert.** force push로 reset하면 다른 사람의 작업이 망가집니다.

## tracking branch (원격 추적 브랜치)
```bash
git branch -vv                    # 각 로컬 브랜치가 어떤 원격 브랜치를 추적하는지
git push -u origin feature        # 처음 푸시 시 -u로 추적 관계 설정
git branch --set-upstream-to=origin/main  # 사후에 추적 관계 변경
```

`origin/main`은 원격 main의 마지막으로 fetch된 상태를 가리키는 로컬 참조입니다. `git fetch`는 이 참조만 갱신하고, `git pull = fetch + merge`입니다.

## HEAD, origin, upstream 차이
- **HEAD**: 현재 체크아웃된 브랜치(또는 커밋)를 가리키는 포인터
- **origin**: 기본 원격 저장소의 별칭 (관습)
- **upstream**: 보통 fork한 원본 저장소를 가리키는 별칭으로 추가 (`git remote add upstream <url>`)
- `origin/main` ≠ `main`: 전자는 원격 추적 브랜치, 후자는 로컬 브랜치

## 흔한 실수 복구
```bash
# 직전 커밋에 빠뜨린 파일 추가
git add 빠뜨린파일
git commit --amend --no-edit

# 잘못된 브랜치에 커밋한 경우
git log -1                # 커밋 해시 확인
git reset --hard HEAD~1   # 현재 브랜치에서 제거
git checkout 올바른브랜치
git cherry-pick <해시>    # 올바른 브랜치로 옮기기

# 푸시한 커밋의 메시지를 바꾸고 싶다 (혼자 쓰는 브랜치인 경우만)
git commit --amend
git push --force-with-lease   # --force보다 안전
```
